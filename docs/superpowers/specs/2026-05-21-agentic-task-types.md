# Agentic Workflow Task Types

## Overview

There are two structurally distinct families of agentic workflow tasks. The distinction is not about difficulty or domain — it is about the nature of ground truth and how validation is performed.

- **Type A — Objective Tasks:** a fixed, verifiable target exists before the agent runs.
- **Type B — Generative Tasks:** ground truth is created by the agent at runtime alongside the output.

This matters architecturally because the evaluation loop, the validator type, and the dataset strategy differ fundamentally between the two.

---

## Type A: Objective / Verifiable Tasks

### What it is

The task has a known, static target. The agent's job is to find or produce something that satisfies hard-coded pass/fail criteria. The target does not change between runs.

### Validation mechanism

Validation is **deterministic and external** to the language model. The agent's output is passed to a tool that has no opinion — it just checks:

- Compiler / linter (code correctness)
- Unit test runner (behavioral correctness)
- Math evaluator / symbolic solver (numeric correctness)
- Regex or schema validator (structural correctness)
- Diff against a reference file (exact match)

If validation fails, the error trace is fed back to the agent as a new context frame and the loop retries.

### Loop diagram

```
  [ Task + Fixed Target ]
          │
          ▼
  [ Agent (generates candidate) ]
          │
          ▼
  [ Deterministic Validator ]
       │         │
    PASS        FAIL
       │         │
       ▼         ▼
  [ Done ]  [ Error → Agent ]
                  │
                  └──────────────────┐
                                     ▼
                          [ Agent (revised candidate) ]
```

### Characteristics

| Attribute | Value |
|---|---|
| Ground truth | Pre-existing, static |
| Validator type | Deterministic external tool |
| Output variance | Low — solution must match fixed criteria |
| Dataset growth | Static; a finite test suite exists |
| Loop termination | Hard pass/fail from the validator |

### Examples

- "Write a Python function that parses this JSON schema and passes all 12 unit tests."
- "Fix the broken SQL migration so that `pytest tests/db/` returns green."
- "Generate a config file that satisfies the JSON Schema at `schema.json`."
- "Refactor `utils.py` so that `mypy --strict` reports zero errors."

## Type B: Generative / Self-Validating Tasks

### What it is

There is no pre-existing target. The agent invents both the output and the criteria for judging it in the same run. Every invocation produces a novel artifact — and with it, a matching evaluation rubric generated from scratch.

This is the pattern behind synthetic data pipelines, educational content generation, adversarial example creation, and randomised test-set construction.

### Why it acts as its own validation dataset

Because the generator produces the ground truth at the same time as the exercise, every run adds a new (question, answer-key) pair to the corpus. There is no fixed dataset to exhaust. Each invocation is simultaneously:

1. A content generation step (the exercise)
2. A ground-truth labelling step (the answer key)
3. An evaluation data point (the pair is reusable for future model assessment)

### Validation mechanism

Validation is **agentic and subjective**. A second LLM node (the Critic/Evaluator) receives the generator's output and scores it against a rubric that was either pre-defined (style, difficulty level, linguistic register) or co-generated by the generator itself.

```
  [ Task Specification ]
  (e.g. "create a B1-level Dutch fill-in-the-blank exercise")
          │
          ▼
  ┌───────────────────────┐
  │  Generator Agent      │  ← invents exercise + answer key together
  └───────────────────────┘
       │              │
  [ Exercise ]   [ Answer Key ]
       │              │
       └──────┬───────┘
              ▼
  ┌───────────────────────┐
  │  Critic Agent         │  ← checks: is it grammatically correct?
  │                       │     is the difficulty calibrated to B1?
  │                       │     does the answer key solve the exercise?
  └───────────────────────┘
       │              │
    PASS            FAIL / revise
       │              │
       ▼              ▼
  [ Serve to user ]  [ Regenerate or patch ]
```

### Characteristics

| Attribute | Value |
|---|---|
| Ground truth | Created at runtime by the generator |
| Validator type | Agentic LLM critic applying a scoring rubric |
| Output variance | High — every run is a different artifact |
| Dataset growth | Unbounded; each run appends a new (input, label) pair |
| Loop termination | Critic score exceeds threshold, or max retries hit |

### Why the Generator must not self-grade

If the same model instance that generated the exercise also evaluates it without separation, it will rationalise its own output. The two roles must be isolated:

- **Different system prompts** (Generator: "be creative"; Critic: "be adversarial and precise")
- **Optionally different model instances** or at minimum a hard context boundary between generation and evaluation steps

### Examples

- "Create a randomised Dutch fill-in-the-blank sentence for a B1 learner, with the correct answer and a distractor list."
- "Generate a novel SQL injection test case with the expected safe/unsafe classification."
- "Produce a logic puzzle at difficulty level 3 with a step-by-step solution."
- "Write a synthetic customer support ticket in the tone of an angry enterprise user, plus the ideal agent response."

---

## Structural Comparison

| Feature | Type A: Objective / Verifiable | Type B: Generative / Self-Validating |
|---|---|---|
| Ground truth | Pre-existing, fixed | Created at runtime |
| Validator type | Deterministic (compiler, test runner, schema) | Agentic LLM critic |
| Output variance | Low | High |
| Dataset role | Consumer of a fixed test suite | Producer of new (input, label) pairs |
| Loop termination signal | Hard binary from external tool | Soft score from critic agent |
| Primary failure mode | Agent cannot satisfy hard constraints | Critic and generator collude / reinforce errors |
| Example domain | Code generation, bug fixing, config synthesis | Education, synthetic data, adversarial examples |

## Hybrid Pattern

Some tasks have a Type A outer shell with a Type B inner component. Example:

> "Generate 50 novel Dutch exercises (Type B) and verify each one is syntactically valid Dutch using a spaCy NLP check (Type A gate)."

The generator creates freely; a deterministic gate filters out malformed outputs before they reach the user. This is the recommended architecture for production synthetic data pipelines — use an LLM critic for semantic quality and a deterministic validator for structural correctness.

```
  [ Generator Agent ]
          │
          ▼
  [ Deterministic Gate ]  ← spaCy parse, schema check, regex
       │         │
    PASS        FAIL → discard / retry
       │
       ▼
  [ LLM Critic ]  ← difficulty calibration, pedagogical quality
       │         │
    PASS        FAIL → regenerate
       │
       ▼
  [ Approved Output ]
```

---

## Implications for AutoAW

| Task type | AutoAW pattern |
|---|---|
| Type A | Fixed benchmark in `datasets/`; agent loop with deterministic validator node; reward = binary pass/fail |
| Type B | No static dataset required; each experiment run extends the evaluation corpus; reward = critic score (float) |
| Hybrid | Gate chain: deterministic filter first, LLM critic second; dataset grows only from PASS outputs |

When designing a new experiment in AutoAW, the first architectural decision is which type the task is. This determines whether to build a static benchmark or a generative pipeline, and whether the validation node should be a tool call or an agent node.
